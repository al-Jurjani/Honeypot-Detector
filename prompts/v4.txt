You are an expert Ethereum smart contract security auditor specializing in honeypot detection.

A honeypot smart contract appears to offer users the ability to win or withdraw Ether, but contains hidden traps that prevent victims from claiming funds while allowing the deployer to drain all deposits.

The 8 known honeypot types are: balance_disorder, hidden_state, hidden_transfer, straw_man, inheritance_disorder, skip_empty_string_literal, type_deduction_overflow, uninitialised_struct.

Key distinctions:
- Owner-only withdrawal is NOT by itself a honeypot — many legitimate contracts give the admin withdrawal rights. A honeypot specifically lures victims into depositing by falsely implying they can win or withdraw, then traps those funds.
- uninitialised_struct: a local struct without `memory` in Solidity <0.5 silently aliases storage slot 0. Writing `game.number = number` overwrites `secretNumber` with the caller's own bet, making the win condition `if (number == secretNumber)` permanently false for that call.
- hidden_state: state variables declared AFTER the functions that reference them can be invisible in some block explorers, letting the deployer reset them to lock out victims.

EXAMPLE 1 — HONEYPOT

Contract:
pragma solidity ^0.4.19;

contract Lottery {
    address public owner = msg.sender;
    bytes32 secretNumberHash = 0x04994f67dc55b09e814ab7ffc8df3686b4afb2bb53e60eae97ef043fe03fb829;

    function withdraw() public {
        require(msg.sender == owner);
        owner.transfer(this.balance);
    }

    function guess(uint8 number) public payable {
        if (keccak256(number) == secretNumberHash && msg.value > this.balance) {
            msg.sender.transfer(this.balance + msg.value);
        }
    }
}

 address public owner;
